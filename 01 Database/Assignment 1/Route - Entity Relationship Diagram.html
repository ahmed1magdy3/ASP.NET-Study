What is an ERD?

Definition

An Entity-Relationship Diagram (ERD) visually represents the data and relationships in a system.

Why use ERDs?

  • Clarify system requirements before building the database
  • Make sure all data is recorded correctly
  • Share the design in a clear way with developers and analysts

Example: "Each employee can work on multiple projects, and each project has many employees."

Why Good Database Design Matters?

Performance (Speed)

  • Fast query execution
  • Data is retrieved easily
  • Scalability for growth
  • Reduces server load

Data Integrity (Correct Data)

  • Accurate information
  • No duplicate data
  • Clear and consistent relationships
  • Enforce business rules

Why Good Database Design Matters? (cont.)

Maintainability (Easy to Update)

  • Easy to update
  • Simple to understand
  • Reduced development time
  • Reduces maintenance problems

User Experience

  • Faster response times
  • Reliable operations
  • Accurate results
  • Smooth functionality
  • Better overall experience

ERD Components

Main Components

  • Entities → Objects (Employee, Project)
  • Attributes → Properties (Name, StartDate)
  • Relationships → Connections (Works On)

Visual Elements

  • Rectangle → Entity
  • Oval → Attribute
  • Diamond → Relationship
Entity Attribute Relation

Entities

Entity: is a distinct and identifiable object in the real world or business domain about which data needs to be stored in a database

Examples:

  • Physical → Employee, Customer, Product
  • Conceptual → Order, Invoice, Reservation

💡 Remember: Each entity will become a table in the database.

Types of Entities

Strong Entity

Can exist independently

Examples: Employee, Product

Weak Entity

Depends on another entity

Examples: Invoice Item depends on Invoice

Attributes

Attributes: is a property, characteristic, or piece of information that describes an entity or a relationship.

In other words — An attribute represents a single type of data that we store about an entity (or sometimes about a relationship).

Types of Attributes

  • Simple: Single-valued (ProductName, Price)
  • Composite: Can be divided (Address → Street, City, ZipCode)
  • Multivalued: Many values (Skills, Languages)
  • Complex: Composite + Multivalued (Dependents → (Name, BirthDate) for multiple)
  • Derived: Calculated (TotalPrice from Quantity × UnitPrice)

Example: Employee(EmployeeID, Name, HireDate, Address, Skills)

Simple Attribute

Definition:

A Simple Attribute (also called Atomic Attribute) is an attribute that cannot be divided into smaller meaningful parts.

It represents a single data value for each entity instance.

➡️ ProductName and Price cannot be meaningfully broken into smaller parts — each represents one simple value.

Composite Attribute

Definition:

A Composite Attribute is an attribute that can be divided into smaller sub-parts, each of which represents meaningful data.

It's a group of related attributes.

➡️ Address and FullName attribute can be split into multiple simple attributes.

Multivalued Attribute

Definition:

A Multivalued Attribute is an attribute that can have more than one value for a single entity instance.

➡️ One customer can have many phones — so Phone is multivalued.

Complex Attribute

Definition:

A Complex Attribute is a combination of Composite and Multivalued attributes.

It means an attribute can have multiple sets of composite values.

➡️ Address is multivalued (many addresses) AND each address has multiple components (street, city, zipcode).

✅ Notes:

  • Rarely used directly in relational models — usually converted into a separate entity.
  • Better approach: Create Address entity related to Customer.
  • Useful in conceptual modeling before normalization.
  • Implementation: Typically becomes a separate table with foreign key to parent entity.

Derived Attribute

Definition:

A Derived Attribute is an attribute whose value can be calculated or derived from other attributes.

It is not stored directly in the database but is computed when needed.

Example:

Entity Derived Attribute Derived From
Product TotalPrice Quantity × UnitPrice
Employee Age Derived from DateOfBirth
Student GPA Computed from grades

✅ Notes:

  • Used for attributes that change frequently or can be recomputed.
  • Reduces redundancy but may increase computation time.
  • Decision: Store vs Calculate depends on performance needs.
  • Example: Age changes every year, better to calculate from BirthDate.

🔑 Identifiers / Keys

Keys uniquely identify records.

Primary Key (PK)

Unique identifier (EmployeeID)

Candidate Key

Possible unique keys (Email, PassportNumber)

Alternate Key

Candidate keys not chosen as PK

Composite Key

Multiple fields (FlightNumber + Date)

Foreign Key (FK)

Links entities (DepartmentID in Employee)

Super Key

Any unique combination (EmployeeID + Name)

Rule: Every entity must have a unique identifier.

Primary Key (PK)

The Primary Key uniquely identifies each record in a table.

  • Each table can have only one primary key.
  • It ensures uniqueness and no NULL values.
  • Automatically creates an index for faster searching.

Example: Employee(EmployeeID, FullName, Email)

PK: EmployeeID

Rule: Every table must have exactly one Primary Key.

Candidate Key

A Candidate Key is any attribute (or combination) that can uniquely identify records.

  • Each candidate key is a potential primary key.
  • There can be multiple candidate keys in one table.
  • Chosen one becomes the Primary Key.

Example: Employee(EmployeeID, Email, PassportNumber)

Candidate Keys: EmployeeID, Email, PassportNumber

Alternate Key

An Alternate Key is a candidate key that was not selected as the primary key.

  • Still maintains uniqueness.
  • Used to enforce business rules (like unique Email).
  • Can be used for lookup or validation.

Example: Employee(EmployeeID, Email)

PK: EmployeeID

Alternate Key: Email

Composite Key

A Composite Key combines two or more fields to uniquely identify a record.

  • Used when no single field can guarantee uniqueness.
  • All parts of the key together ensure uniqueness.
  • Common in junction tables (many-to-many relationships).

Example: Enrollment(StudentID, CourseID, Semester)

Composite Key: (StudentID + CourseID)

Foreign Key (FK)

A Foreign Key links one table to another using a primary key from the parent table.

  • Ensures referential integrity between tables.
  • Prevents orphan records (child must match parent).
  • Defines relationships such as One-to-Many or Many-to-One.

Example: Employee(EmployeeID, DepartmentID)

FK: DepartmentID → Department(DepartmentID)

Super Key

A Super Key is any set of attributes that can uniquely identify a record — may include extra fields.

  • Every Candidate Key is a Super Key.
  • But not every Super Key is minimal.
  • Used in conceptual design to test uniqueness combinations.

Example: (EmployeeID + Name) or (Email)

Super Keys: EmployeeID, (EmployeeID + Name), (Email + Phone)

Relationship Types

Relationships describe how entities are connected in an ERD. They can involve one, two, or more entity types.

Unary (Recursive)

A relationship between instances of the same entity.

  • Also called self-relationship.
  • Example: An Employee supervises another Employee.

Binary

A relationship between two different entities.

  • Most common type in ERDs.
  • Example: Customer places an Order.

Ternary

A relationship involving three different entities.

  • Used when a single relationship depends on three entities together.
  • Example: Doctor prescribes a Medicine to a Patient.

Tip: The type of relationship depends on how many entity sets participate in it.

Cardinality

Cardinality defines the number of instances of one entity that can or must be associated with instances of another entity.

  • 1:1 (One-to-One) — Each entity in A is related to only one in B.
    Example: Each Person has one Passport.
  • 1:M (One-to-Many) — One record in A can relate to many in B.
    Example: One Department has many Employees.
  • M:N (Many-to-Many) — Many records in A can relate to many in B.
    Example: A Student can enroll in many Courses, and each course has many students.

Rule: Cardinality defines how many entities are related.

Participation

Participation shows whether all or only some entity instances are involved in a relationship.

  • Total Participation — Every instance must participate in the relationship.
    Example: Every Invoice must belong to a Customer.
  • Partial Participation — Some instances may or may not participate.
    Example: Not every Employee is assigned to a Project.

💡 Important: Participation defines whether entities must take part in the relationship.

Identifying vs. Non-Identifying

Identifying Relationship

Child depends on parent (Weak entity).

Example: Invoice → InvoiceItem

Non-Identifying

Child exists independently.

Example: Department → Employee

Common Mistakes in ERD Design

  • 🚫 Treating attributes as entities
  • 🚫 Ignoring many-to-many relationships
  • 🚫 Missing identifiers
  • 🚫 Using inconsistent names

✅ Best Practices

  • ✓ Always name relationships meaningfully
  • ✓ Double-check cardinality with business logic

Weak Entities (Deep Dive)

  • Depend on another entity for identity
  • Cannot exist without parent entity
  • Represented by double rectangle and double diamond
  • Partial key: identifies within relationship

Example: Invoice → InvoiceItem

InvoiceItem has composite key (InvoiceID + ItemNumber)

Specialization (Top-Down)

Defines subclasses based on characteristics

"Employee can be Manager or Developer"

Parent: Employee

Sub-entities: Manager, Developer

Key Characteristics

  • Superclass: The general entity (e.g., Employee).
  • Subclass: The specialized entity (e.g., Manager, Developer).
  • Attribute Inheritance: Subclasses inherit all attributes of the superclass and may have additional ones.

Types:

  • Disjoint (D): An entity instance can belong to only one subclass.
    Example: An employee cannot be both a manager and a developer.
  • Overlapping (O): An entity instance can belong to multiple subclasses.
    Example: An employee can be both a trainer and a developer.

Generalization (Bottom-Up)

Combine similar entities into general entity

Example: Car, Truck → Vehicle

Key Characteristics

  • Bottom-Up Process: Starts by finding common features among entities.
  • Superclass: The newly created, more general entity (e.g., Vehicle).
  • Shared Attributes: Common properties are moved to the superclass.
  • Unique Attributes: Remain in the original sub-entities if still needed.

Purpose:

  • Reduce redundancy
  • Simplify relationships

Aggregation (Has-A)

Relationship participates in another relationship

Key Characteristics

  • Shows a “Has–A” or “Part–Of” relationship between a relationship and an entity.
  • Used when a relationship itself needs to participate in another relationship.
  • Represents a higher-level abstraction — treating the relationship as an entity.

Example: "Project employs Employees and uses Equipment"

Notation:

Diamond inside rectangle

💡: Relationship treated as entity.

ERD Design Process

  1. Identify entities (nouns)
  2. Identify attributes (details)
  3. Identify relationships (verbs)
  4. Determine cardinalities
  5. Identify keys (PK, FK)
  6. Apply advanced concepts
  7. Validate with stakeholders

ERD Best Practices

✅ Checklist

  • ✓ Use meaningful names
  • ✓ Avoid M:N without bridge entities
  • ✓ Keep attributes atomic
  • ✓ Every entity has primary key
  • ✓ Use consistent naming
  • ✓ Document assumptions

🚫 Pitfalls

  • 🚫 Ignoring derived attributes
  • 🚫 Wrong primary keys
  • 🚫 Missing relationships
  • 🚫 Overcomplicating diagrams [Adding unnecessary details]
